1️⃣ IDOR (Insecure Direct Object References) - FIXED ✅

What is IDOR?

Users can access or modify objects (quizzes, results) they don't own by changing IDs in the URL/request.

Test Scenario: Two instructors, one quiz

Setup:

  • instructor1 creates Quiz ID: 1
  • instructor2 tries to modify it (should FAIL)

Test 1: instructor2 tries to UPDATE instructor1's quiz

Request:

PUT /api/quiz/1
Authorization: instructor2
Body: {"title": "HACKED"}

Expected Response:

{
"error": "Unauthorized - not quiz owner"
}
HTTP Status: 403 Forbidden

Result:

✅ PASS - Access Denied

Test 2: instructor2 tries to DELETE instructor1's quiz

Request:

DELETE /api/quiz/1
Authorization: instructor2

Response: 403 Forbidden

✅ PASS - Deletion Denied

Test 3: instructor2 tries to VIEW instructor1's results

Request:

GET /api/quiz/1/results
Authorization: instructor2

Response: 403 Forbidden

✅ PASS - View Denied

How We Fixed It:

# In database, every quiz has creator_id
def check_quiz_ownership(quiz_id, user_id):
  quiz = Quiz.query.get(quiz_id)
  if quiz.creator_id != user_id: # CHECK OWNERSHIP
    return "Unauthorized", 403
  return quiz, 200

# Applied to: PUT, DELETE, GET /results

2️⃣ CSRF (Cross-Site Request Forgery) - FIXED ✅

What is CSRF?

Attacker's website can forge requests that appear to come from the user, bypassing session authentication.

Attack Scenario (Prevented)

Without protection:

<!-- Attacker's website -->
<form action="https://quiz.app/api/quiz/1" method="POST">
  <input name="title" value="HACKED">
</form>
<!-- User unknowingly submits this form -->

Security Headers Check

Request:

curl -I http://localhost:5000/

Response Headers:

Set-Cookie: session=...; HttpOnly; Path=/; SameSite=Strict; Expires=...
Cookie Flag Status Purpose
HttpOnly ✅ Present JavaScript cannot access or steal cookie
SameSite=Strict ✅ Present Cookie NOT sent to cross-site requests
Secure ✅ Production Cookie only sent over HTTPS

How We Fixed It:

# SameSite=Strict prevents cross-site cookie transmission
app.config['SESSION_COOKIE_SAMESITE'] = 'Strict'
app.config['SESSION_COOKIE_HTTPONLY'] = True

# Browser behavior:
# Request from attacker.com -> Cookie NOT sent
# Forged form fails because no session = no authentication
✅ PASS - CSRF Protection Active

3️⃣ Clickjacking - FIXED ✅

What is Clickjacking?

Attacker embeds your application in a hidden iframe and tricks users into clicking invisible buttons.

Attack Scenario (Prevented)

Without protection:

<!-- Attacker's website -->
<iframe src="https://quiz.app" style="opacity:0"></iframe>
<!-- User clicks what appears to be attacker's button -->
<!-- But actually clicks quiz button -->

Protection Headers Verification

Response Headers:

X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none'
X-Content-Type-Options: nosniff
X-XSS-Protection: 1; mode=block
Header Status Effect
X-Frame-Options: DENY ✅ Present Browser refuses to load in iframe
CSP frame-ancestors ✅ Present Modern CSP protection (same effect)
X-Content-Type-Options ✅ Present Prevents MIME sniffing attacks

How We Fixed It:

@app.after_request
def set_security_headers(response):
  response.headers['X-Frame-Options'] = 'DENY'
  response.headers['CSP'] = "frame-ancestors 'none'"
  return response

# Browser behavior:
# If attacker tries: <iframe src="quiz.app">
# Browser REFUSES to load the page
✅ PASS - Clickjacking Protection Active
